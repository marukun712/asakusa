# Context
Webは本来、文書を出版・リンクすることを目的としたナレッジベースプロトコルである。
URLは文書の場所を指し示すために存在し、DNSも文書の場所をより人間に読みやすくする役割を持つ。
Staticなものを取り扱うとき、Webは最大のパフォーマンスを発揮できる。

ソーシャルメディアの登場以降、WebはDynamicなものを扱うようになった。
たとえば、人のアイデンティティ、動的に生成されるページ etc...

Webの基本単位が文書から人に移り変わろうとしている。

Web・コンピューター世界において、エンティティを指し示すためにはURLやIDといったポインタが必須である。しかし、人にポインタづけをすることはとても難しい。

公開鍵は人ではなく文書に署名をするためのStaticなハンコであるし、IDはそのポインタ自体には意味を持たない値である。そしてその公開鍵・IDと人を紐付けるためには、第三者による信頼グラフや、中央管理者への信頼など、様々な追加レイヤーを必要とする。

しかし、揮発性のコミュニケーション、VCのような、同期的な文脈を持つ「場」において、アイデンティティはその場の他者からの観測によって、それぞれの観測者の脳内に形成される。ここで人を指し示すポインタは一切使われていない。

このようなDynamicな揮発性の「場」こそが、デジタル空間において人間のアイデンティティを確立させるもっとも適した基盤であると考えている。

Polka Protocolは、Dynamicなアイデンティティ基盤としての「場」と、Staticな文書基盤としての「Web」という2つの役割を明文化する。

Dynamicな場では、その場で形成されたアイデンティティと、公開鍵というハンコを結びつける役割を担い、Staticな場ではすでに成立したアイデンティティの上で、人を介入させない、文書を基本単位としたナレッジネットワークを構築する。

# Static Part

Polka Protocol Static Partは、Geminiプロトコルを一部踏襲し、暗号学的な純粋さを追求したナレッジベースプロトコルである。

テキストのマークアップには、Gemini Protocolのgemtextフォーマットを用いる。

トランスポートにはTCPを用いる。

通信の暗号化・中間者攻撃の防止には、Noise Protocol Frameworkを用いる。
本プロトコルにおいて、この人間がこの公開鍵というハンコをつかっている、ということはDynamic Partによって達成されるため、Static Partでは鍵の信頼問題を問題としない。

公開鍵はフィンガープリント・DiceBearによる決定論的なアバターの2つで表現される。

Polka Document Serverは、あるフォルダ配下のファイルを公開する純粋なStatic File Serverである。ただし、このフォルダ以下のドキュメントのhash値でMerkle Treeを構成し、そのルートハッシュに対して、署名鍵で署名をする。

この形式を取ることで、サーバー状態全体の完全性を単一の署名で証明することができる。

検索可能性のための分散型イベントストリームを設ける。
このイベントストリームはNostr Relayのように、誰でも建てることができる。

Document Serverは、必要に応じて任意のリレーにドキュメントの追加・変更されたドキュメントのURLを広告することができる。

インデクサーは、このリレーからの情報をもとに独自の実装でインデックスを組み立てることができる。バックリンク構築もこのインデクサーが行う。

見知らぬ相手には、そもそも人物的なイメージが存在しないため、暗号鍵による一貫性のみが確認できるTOFUで十分である。

# Dynamic Part

Polka Protocol Dynamic Partは、揮発性のコミュニケーションを実現するための純粋な音声会話サーバーである。

トランスポートにはUDPを用いる。

音声会話サーバーに、Opusエンコードした音声を送信する。
音声会話サーバーは、受信した音声を単純にブロードキャストする。

音声には、一時的に生成された公開鍵による署名を添付する。

公開鍵はフィンガープリント・DiceBearによる決定論的なアバターの2つで表現される。

この鍵は、セッション内での一貫性を確保することが目的であり、このセッション中の用途以外には一切仕様されない。

ここではDynamic Partを純粋な音声サーバーとして定義はするものの、物理的な空間、Discord、VRChatなど、他者からの観測が生まれるすべての揮発性コミュニケーションの「場」をDynamic Partとして定義する。

そのため本プロトコルでは、最低限の自己主権的にホストできる簡易音声サーバーの仕様のみが定義されている。この実装は小規模想定のため、通信の暗号化についてはネットワークレイヤに委譲するものとする。

# 公開範囲に対する考え方

Static PartとDynamic Partの両方において、公開範囲をコントロールしたい場合が考えられる。

Polka Protocolは、公開範囲のコントロールをネットワークレイヤに委譲する。ネットワークの到達可能範囲が公開範囲である。

WireGuard等を用いて公開範囲をコントロールすることを推奨する。

# 実装ガイド

ディレクトリ

docs/ ...文書公開サーバー
relay/ ...UDPブロードキャストを用いたリレーサーバー
space/ ...Dynamic Partの実装。Opusコーデックでエンコードされた音声を受け取り、UDPブロードキャストするサーバー

tools/setup ...セットアップスクリプト(２つのkeypairを生成)
tools/builder ...docsサーバーにアップロードするデータセットのMerkle Rootをとり、秘密鍵をパスワードから復号して署名

鍵の扱い

トランスポート層の暗号化については、NoiseのNXパターンを用いる
Noise用のkeypairと、Merkle用のkeypairは別のものを用いる
Merkle用のkeypairはageで暗号化する(セットアップスクリプトでパスワードを求める)

~/.polka/transport/alg(秘密鍵) alg.pub
~/.polka/content/alg.age alg.pub

検証

.well-known/polkaに、
content-pk.root-hash.timestamp.sigという形式で置き、クライアントはそれを検証

ここでクライアント側は、フィンガープリントやDiceBearアバターを導出し目視チェック

すべて、denoで管理します。deno monorepoで管理します。
