# Polka Protocol - 背景と動機

Webは本来、文書を出版・リンクすることを目的としたナレッジベースプロトコルとして設計されました。URLは文書の場所を指し示し、DNSによってホスト名を人間が読みやすい形で扱えます。静的なコンテンツを扱うとき、Webはその設計の本領を発揮します。

ソーシャルメディアの登場以降、Webは人のアイデンティティや動的に生成されるコンテンツを扱うようになり、基本単位が「文書」から「人」へと移りつつあります。

デジタル空間でエンティティを指し示すには、URLやIDといったポインタが必要です。しかし、人にポインタを割り当てることは本質的に困難です。公開鍵は人ではなく文書に署名するための静的な印鑑であり、IDはそれ自体ではアイデンティティを持たない値です。公開鍵・IDと人を結びつけるためには、信頼グラフや中央管理者への依存など、追加のレイヤーが必要になります。

一方、音声通話のような同期的な揮発性コミュニケーションの「場」では、アイデンティティはその場にいる他者からの観測によって各人の脳内に自然と形成されます。ここではポインタは使われません。

**Polka Protocolは、この揮発性の「場」をアイデンティティ基盤として位置づけ、その上に静的な文書ネットワークを構築するための2層プロトコルです。**

Dynamic Part(場)では、場において形成されたアイデンティティと公開鍵を結びつけます。Static Part(文書)では、確立されたアイデンティティの上で、人を介在させない文書を基本単位としたナレッジネットワークを運営します。
