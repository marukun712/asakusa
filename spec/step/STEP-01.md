# STEP-01: コア仕様

## 1. Static Part

### 1.1 概要

Static Partは、Gemini Protocolを参考にしつつ、暗号学的な純粋さを追求したナレッジベースプロトコルです。テキストのマークアップにはGemini Protocolのgemtextフォーマットを使用します。

### 1.2 トランスポート

- プロトコル: TCP
- 暗号化: Noise Protocol Framework (NXパターン)

Noise NXパターンでは、サーバーが静的鍵ペアを持ち、クライアントは匿名で接続します。これは「誰でも読める文書サーバー」の性質と一致します。

本プロトコルにおいて、「この公開鍵を持つ人物が誰であるか」の解決はDynamic Partの責務です。Static Partでは鍵の人物的信頼を問題としません。初回接続時はTOFU(Trust On First Use)で十分です。

### 1.3 公開鍵の表現

公開鍵は以下の2つの形式で表現されます:

- **フィンガープリント**: 公開鍵のSHA-256ハッシュの16進数表現
- **視覚的アバター**: 公開鍵から決定論的に生成された視覚的表現

これらはクライアントが目視でサーバーのアイデンティティを確認するために用います。

### 1.4 リクエスト/レスポンス形式

リクエストおよびレスポンスはGemini Protocolの仕様に準拠します。

https://geminiprotocol.net/docs/protocol-specification.gmi

### 1.5 Document Server

Document Serverは、指定したディレクトリ以下のファイルを提供する静的ファイルサーバーです。

加えて、ディレクトリ以下の全ファイルのSHA-256ハッシュでMerkle Treeを構成し、そのルートハッシュにコンテンツ鍵で署名します。これにより、サーバーが公開するコンテンツ全体の完全性を単一の署名で証明できます。

**検証情報の配置**:

```
/.well-known/polka
```

フォーマット:

```
<content-pk>.<root-hash>.<timestamp>.<sig>
```

各フィールドは16進数エンコードされた値、`timestamp`はUnixミリ秒(10進数)です。

クライアントはこのファイルを取得し、署名を検証します。その後、公開鍵からフィンガープリントと視覚的アバターを導出し、目視でサーバーのアイデンティティを確認します。

### 1.6 分散型イベントストリーム (Relay)

Document Serverは任意のRelayサーバーにドキュメントの追加・更新を広告できます。Relayは誰でも自由に立て、独立して運営できます。

**Relay通信**: UDP

**イベント形式** (JSON):

```typescript
{ "type": "add" | "update", "url": "<document-url>" }
```

Relayは受信したイベントを他の全参加者にUDPブロードキャストします。インデクサーはRelayからの情報をもとに独自のインデックスを構築できます。バックリンクの構築もインデクサーの責務です。

### 1.7 鍵管理

2種類の鍵ペアを使用します:

| 用途 | 種類 | 保存場所 |
|------|------|---------|
| トランスポート暗号化 | X25519 (Noise NX用) | `~/.polka/transport/key`, `key.pub` |
| コンテンツ署名 | Ed25519 | `~/.polka/content/key`, `key.pub` |

コンテンツ署名鍵はパスフレーズで暗号化して保管します。

## 2. Dynamic Part

### 2.1 概要

Dynamic Partは、揮発性コミュニケーションの「場」を提供するプロトコルです。本仕様では、誰でも自己ホストできる最小限の音声サーバーを定義します。

物理的な空間、Discord、VRChatなど、他者からの観測が生まれるすべての揮発性コミュニケーションの「場」をDynamic Partとして定義します。本プロトコルが実装として提供するのはそのうちの最小形態です。

### 2.2 トランスポート

- プロトコル: UDP
- 暗号化: ネットワークレイヤに委譲 (WireGuard等を推奨)

小規模運用を前提とするため、アプリケーションレイヤでの暗号化は実装しません。

### 2.3 音声フォーマット

クライアントはOpusコーデックでエンコードした音声をUDPパケットとして送信します。サーバーは受信したパケットを他の全参加者にそのままブロードキャストします。エンコード・デコードはクライアントの責務です。

### 2.4 セッション内アイデンティティ

各クライアントはセッション参加時に一時的な鍵ペアを生成し、送信するパケットに署名を添付します。

この鍵はセッション内での発言の一貫性を証明するためのものであり、セッション終了後は破棄されます。公開鍵はフィンガープリントと視覚的アバターで表現されます。

署名の検証はクライアントの責務です。サーバーは検証を行いません。

---

## 3. 公開範囲

Static Part・Dynamic Part ともに、公開範囲の制御はネットワークレイヤに委譲します。**ネットワークの到達可能範囲が公開範囲です。**

アクセス制御が必要な場合はWireGuard等のVPNを用いてネットワーク自体を閉じることを推奨します。
